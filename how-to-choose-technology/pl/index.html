W czym więc problem?

  • projektów realizowanych solo

Szacowanie / ocenianie

Jakich wyborów musimy dokonywać?

  1. język programowania
  2. framework
  3. biblioteki/zależności
  4. typ bazy danych
  5. silnik bazy danych
  6. platforma docelowa
  7. chmura
  8. codefull / no-code

Kwestie do rozwiązania

  • jak ten język programowania wpłynie na produktywność zespołu?
  • czy ten framework przyczyni się do poprawy jakości i tempa prac?
  • czy ta biblioteka jest godna zaufania?
  • czy ten silnik obsłuży rozmiar moich transakcji?
  • czy ta platforma pozwala mi dostatecznie konfigurować środowisko?
  • czy ta chmura oferuje wystarczająco dużo zasobów do skalowania?

Popularność

  • ilość gwiazdek na Github
  • szerokie community
  • łatwe wsparcie
  • łatwa rekrutacja

Rozwój

  • czas trwania aktywnego rozwoju
  • ilość issues Github
  • most-upvoted issues
  • czas rozwiązywania issues

Jakość

  • czy projekt ma testy
  • pokrycie testami
  • statyczna analiza kodu
  • skany security

Prostota

  • czas poświęcony na implementację
  • próg wejścia
  • good enough vs kombajn

Uzależnienie

  • separacja logiki biznesowej od frameworka
  • vendor lock
  • abstrakcja zależności
  • fallback solution

Swoboda działania

  • framework/lib vs use cases
  • łatwość implementacji niestandardowych use cases
  • znikoma koneczność "hackowania"

Jak ocenić te wszystkie "metryki"?

  1. doświadczenie
  2. wsparcie zespołu
  3. POC - proof of concept
  4. evaluation period
  5. korekta błednych decyzji

Relacja z biznesem

  • programiści tworzą
  • ktoś inny za to płaci
  • ten kto płaci, chce efektów jak najszybciej
  • programiści chcą jakości

Decyzje projektowe muszą być klarowne, powody logiczne, a efekty namacalne

  • decyzje podejmujcie całym zespołem
  • każdy wnosi swoje doświadczenie i obawy
  • błędy na etapie projektu są najłatwiejsze do naprawy
